FRAMING

PROBLEM STATEMENT:

The Jobs vertical needs to be integrated into the General Classifieds platform to unify KSL's marketplace products and modernize the underlying technology. The legacy Jobs infrastructure carries significant technical debt, risks $1.2M in annual revenue, and blocks the future Homes vertical migration.

Phase 2 specifically addresses the user-facing layers: Search Results Page (SRP), listing Detail Page, and MyAccount — the pieces needed to make Jobs listings fully discoverable, viewable, and manageable by users on the new platform.

POTENTIAL SOLUTION/S:

Build the backend infrastructure to support Jobs on the SRP, Detail Page, and MyAccount by:

  • Replacing the legacy subcategory-driven specs model with a semantic facets/filters system (e.g., jobtype=parttime)
  • Syncing Jobs data through Mongo connectors to Elasticsearch
  • Providing GraphQL endpoints for SRP, Detail, and MyAccount
  • Implementing soft delete so Customer Experience can restore listings
  • Supporting app-side rendering of Jobs-specific fields in PAL, SRP, and Detail

APPETITE & COST:

Web: 4 weeks | App: 2 weeks

  • Platform Modernization: Eliminates legacy PHP infrastructure, consolidates tech stack, and enables future Homes vertical migration without duplicate work

SHAPING

IDENTIFIED SOLUTION SUMMARY (TL;DR)

This phase delivers three main pieces: the Search Results Page, the listing Detail Page, and a MyAccount area where people can manage their listings. We are also updating how listing specs and filters work — replacing subcategory-specific rules and hard-to-read IDs in the URL with clear, meaningful filters (e.g., jobtype=parttime). When someone browses Jobs, they will see job-related filters like salary range, hourly rate, and job type.

On the SRP, job listings will be included in search and filterable by listing type. For the Detail Page, we'll provide the data needed to display and update a listing. MyAccount will let users view, edit, and delete their existing listings. As this phase deals with front-end components, it will require some frontend resources (TBD).

GOALS

Specifications Architecture Change

  • Replace non-semantic spec identifiers in URLs (e.g., specs/1) with semantic key/value filters (e.g., sex=male)
  • Deliver a facets/filters system that shows Jobs-specific filters (salary range, hourly rate, job type) when Jobs category or listing type is selected
  • Make available filters dynamic based on selected category and listing type, rather than subcategory-dependent
  • Model and store specification definitions with clear parent dependencies (category, listing type, seller type, etc.)
  • Implement an initial config path in code; capture requirements for a future Specifications Manager admin UI

SRP

  • Update Mongo connectors to sync job data to ES
  • Update ES queries to filter by listing type (Classified vs. Jobs) — same index filtering
  • GraphQL data structure for front end
  • Front End Requirements (TBD)

Detail Page

  • Read endpoints with all detail page data needed for Jobs
  • GraphQL endpoint for Detail Page updates

MyAccount

  • View, update, and delete listings (using existing ES queries to view)
  • Add Soft Delete functionality
  • New endpoint for Nest Listing Restore tool

App Work (2-week estimate)

  • SRP: Users select Jobs, opens to pre-selected Jobs subcategories; remove Jobs from global search
  • Detail Page: Hook into pre-existing unified detail page
  • PAL: Render job-specific fields; update to use PAL endpoint for posting/editing; hard-code fields (similar to Cars)

ACCEPTANCE CRITERIA

  • Filters use semantic key/value pairs in URLs (e.g., sex=male); legacy specs/<id> pattern no longer required for new Jobs SRP/filter URLs
  • Available filters are driven by selection context (category, listing type, seller type) rather than subcategory-specific mappings
  • When Jobs is selected, SRP shows Jobs-specific filters including salary range, hourly rate, and job type
  • A data model exists for specification definitions including parent dependencies; Jobs can be represented without special-case behavior
  • SRP search logic previously embedded in GraphQL is implemented in a dedicated service
  • SRP results can be filtered by listing type (Classifieds vs. Jobs) using same-index filtering
  • SRP display rules implemented:
    • Jobs mixed with other listing types → standard grid format
    • Jobs is the only selected category → Jobs list format
    • Jobs + Services selected → standard grid style
    • Jobs available as a listing type option alongside For Sale, In Search Of, and For Rent
  • Detail Page reads for Jobs listings served from new Listing Service endpoints, including all data needed for the unified detail page
  • Detail Page supports updating a listing via the defined GraphQL endpoint(s), validated end-to-end
  • MyAccount allows users to view, edit, and delete listings using the new Listing Service, reusing existing ES queries where appropriate
  • Deleting a listing performs a soft delete (not hard delete); CX can restore without losing photos
  • Soft-delete retention and final purge behavior are documented and agreed (including Legal inputs)

EXPECTED METRIC IMPACT

  • Jobs listings become fully discoverable, filterable, and manageable on the unified Classifieds platform
  • Protects 400K monthly page views and $1.2M annual revenue from legacy platform risk
  • Unblocks Phase 3 (Saved Search & Alerts) and future phases

SOLUTION DETAILS

FEATURE DESCRIPTION

Specifications Architecture Change

  • Replace the legacy, subcategory-driven specs model with a facets/filters system using semantic key/value pairs
  • When a user selects Jobs (by category or listing type), the SRP dynamically surfaces Jobs-specific filters (salary range, hourly rate, job type)
  • Available filters driven by current selection context, not hard-coupled to subcategories
  • Define a clear data model for specification definitions with parent dependencies, expandable to other verticals later
  • Deprecate reliance on Category Manager for specs; start with config in code

Search Results Page (SRP)

  • Move SRP search logic (currently embedded in GraphQL ES query building) into a dedicated service
  • Confirm whether Jobs filters appear immediately when Jobs is selected or only after subcategory selection
  • Define filter grouping and display in collapsible sections
  • Reference: List of Job Facets and Specifications
  • Reference: Figma Mockup of Jobs SRP

Detail Page

  • Migrate Detail Page reads from the current GraphQL-through-CAPI approach to new Listing Service endpoints
  • Maintain legacy compatibility near-term by continuing to store contact info and verification status on listing documents while downstream consumers migrate

MyAccount

  • View, update, and delete listings via new Listing Service, reusing existing ES queries where it makes sense
  • Implement soft delete so CX can reactivate listings without losing photos
  • Define retention period and final purge behavior for soft-deleted listings (requires Legal input)

Technical Implementation Steps

  1. Update Mongo connectors to sync additional Jobs fields to ES
  1. Update AvailableFilters from SRP GraphQL call to include Jobs field filters
  1. Build out SRP search queries functionality in backend (move from GraphQL into dedicated service)
  1. Expand current Detail Page endpoints to send back Jobs-specific fields
