New endpoints and their needed logic

All listings MUST ALWAYS contain a listingType to allow the service to determine the shape of the date.

  • Create ( POST listing/ )
    • The body of the post MUST contain the memberId to whom the listing belongs
  • Update ( PUT listing/{id} )
    • This endpoint MUST match the owner to the caller.
    • Received a partial set of data and updates specific fields in the record.
  • Activate ( PUT listing/{id}/activate )
  • Checkout ( PUT listing/{id}/checkout )
  • Delete Listing ( DELETE listing/{id})
    • Must validate request by
  • Get ( GET listing/{id} )
    • For Detail Page (full details)
  • GetByMember
    • We will possibly need two versions.
      • One for getting just the ID and Stock#, so the feeds system can know which listings to create, update, or delete ( GET/PUT/DELETE listing/partial?memberId={memberId} )
      • The other for pulling the full details the listings ( GET listing?memberId={memberId} )
        • Would this be paginated?
    • For getting canonical source of truth data for internal services.
      • Feeds
      • CX tools
    • For MyAccount
